## 4. 작업용 개인 클라우드, Jisong Cloud 6.0

### # 1) 저장소가 아니라 작업대라는 발상

파일은 계속 생깁니다.

수업 자료를 받으면 PDF가 생기고, 논문을 읽으면 메모가 생기고, 실습 중 정리한 내용은 Markdown으로 남습니다. AI에게 파일을 분석시키면 답변이 생기고, 그 답변을 다시 정리하면 또 다른 문서가 됩니다.

문제는 파일이 많아진다는 것 자체가 아닙니다.

진짜 문제는 흐름이 끊긴다는 것입니다. 파일은 다운로드 폴더에 있고, 메모는 다른 앱에 있고, AI 분석 결과는 채팅창 어딘가에 있고, PDF 변환은 또 다른 사이트에서 해야 합니다. 하나하나는 별것 아닌데, 이 과정이 반복되면 생각보다 많은 에너지가 새어 나갑니다.

어느 순간 이런 생각을 했습니다.

개인용 클라우드는 꼭 파일을 보관하는 곳이어야 할까. 파일을 올려두고, 필요할 때 내려받는 저장소만으로 충분할까. 제가 원한 것은 단순한 저장소가 아니었습니다. 파일을 올리고, 분석하고, 메모하고, 다시 문서로 만들고, 나중에 재사용하는 흐름.

그 흐름을 한 화면 안에서 이어갈 수 있는 개인 작업대가 필요했습니다. Jisong Cloud 6.0은 그 생각에서 시작한 프로젝트입니다.

### # 1) 처음에는 저장소에 가까웠다

Jisong Cloud의 초기 버전은 지금보다 훨씬 단순했습니다.

당시 핵심은 세 가지였습니다.

파일 업로드. 웹 메모장. 도구상자. 파일 업로드는 문서를 서버에 올리고 확인하는 흐름이었습니다. 웹 메모장은 정식 문서가 되기 전의 생각을 빠르게 적어두는 공간이었습니다. 도구상자는 글자 수 계산이나 간단한 선택 보조처럼 반복되는 작은 작업을 모아둔 영역이었습니다.

지금 보면 단순합니다.

하지만 이 단순한 기능들에도 공통점이 있었습니다. 반복되는 행동을 찾고, 입력을 정하고, 조건을 세우고, 출력을 정하는 과정이 먼저 있었습니다. 그러니까 Jisong Cloud v2는 완성된 서비스라기보다, 제가 반복하던 행동을 세 가지 구조로 나눠본 실험에 가까웠습니다. 당시에는 Streamlit과 Python을 사용했습니다.

이유도 단순했습니다.

복잡한 웹 개발 구조를 먼저 세우기보다, 아이디어를 빠르게 화면으로 만들고 싶었기 때문입니다. 파일을 올린다. 메모를 쓴다. 작은 도구를 실행한다.

그 정도의 흐름을 확인하기에는 Streamlit이 충분했습니다. 중요한 것은 기술 자체가 아니었습니다. 제가 어떤 반복을 가지고 있고, 그 반복을 어디까지 구조화할 수 있는지 보는 것이 더 중요했습니다.

### # 2) v6에서 달라진 것은 기능보다 구조다

Jisong Cloud 6.0은 이전 버전과 다릅니다. 가장 큰 차이는 기능이 많아졌다는 것이 아닙니다.

구조가 바뀌었습니다.

초기 버전이 Streamlit으로 빠르게 만든 개인용 실험에 가까웠다면, v6는 Starlette API와 정적 프론트엔드, Google Cloud Storage, Gemini, Cloud Run을 연결한 운영형 구조에 가깝습니다.

브라우저는 화면을 담당합니다. API 서버는 기능 사이의 동선을 연결합니다. GCS는 파일과 메모를 저장합니다. Gemini는 분석을 담당합니다. Cloud Run은 이 작업대를 인터넷 위에서 실행 가능하게 만듭니다.

이 변화는 꽤 중요했습니다.

Streamlit 버전에서는 하나의 앱 안에 기능이 붙어 있는 느낌이 강했습니다. 반면 v6에서는 파일, 메모, AI, 도구가 각각 API endpoint를 통해 연결됩니다. 이렇게 바뀌면 프로젝트를 보는 관점도 달라집니다. 이제 Jisong Cloud는 단순히 화면 하나짜리 앱이 아닙니다.

파일을 올리는 기능, 메모를 관리하는 기능, AI로 분석하는 기능, PDF로 내보내는 기능, 텍스트를 정리하는 기능이 하나의 작업 흐름 안에서 연결된 개인 인프라가 됩니다. v2가 “반복을 발견한 실험”이었다면, v6는 “반복을 운영 가능한 작업대 위에 올린 구조”에 가깝습니다.

### # 3) 파일은 보관물이 아니라 작업의 입구다

보통 클라우드라고 하면 먼저 저장 공간을 떠올립니다. 파일을 업로드한다. 어디서든 접근한다. 필요하면 다운로드한다.

물론 이것도 중요합니다.

하지만 저에게 파일은 보관물이기 전에 작업의 입구입니다. 논문 PDF를 올리면 내용을 요약하고 싶습니다. 강의자료를 올리면 핵심을 뽑고 싶습니다. CSV를 올리면 구조를 보고 싶습니다. Markdown 파일을 올리면 PDF로 변환하고 싶습니다. 이미지를 올리면 설명을 붙이고 싶습니다.

그러니까 파일 업로드는 끝이 아닙니다.

오히려 시작입니다.

Jisong Cloud 6.0에서는 파일을 올린 뒤, 그 파일을 다시 분석하고, 필요하면 결과를 복사하고, Markdown으로 내려받고, PDF로 만들고, 다시 메모로 저장할 수 있습니다. 파일이 저장소 안에 멈춰 있는 것이 아니라, 다음 작업으로 이어집니다. 이것이 제가 원한 개인용 클라우드의 형태였습니다.

파일을 보관하는 공간이 아니라, 파일에서 시작되는 작업 흐름을 붙잡는 공간.

### # 4) 메모는 생각의 중간 저장소다

글을 쓰다 보면 완성된 문서가 되기 전의 문장이 필요합니다. 아직 제목이 없고, 순서도 정해지지 않았고, 그냥 떠오른 생각에 가까운 문장들. 이런 문장은 생각보다 쉽게 사라집니다. 채팅창에 남겨두면 나중에 찾기 어렵고, 로컬 파일로 만들기에는 아직 너무 가볍고, 머릿속에만 두면 거의 반드시 잊어버립니다.

그래서 메모장이 필요했습니다.

Jisong Cloud의 메모장은 거창한 노트 앱이 아닙니다. 생각을 잠깐 붙잡아두는 공간입니다. 메모를 쓰고, 수정하고, 복사하고, 다운로드하고, 삭제할 수 있습니다. 중요한 것은 메모가 AI 분석과 연결된다는 점입니다.

AI 분석 결과를 새 메모로 저장할 수 있고, 저장된 메모는 다시 열어서 다음 작업의 재료로 쓸 수 있습니다.

이 구조가 마음에 듭니다.

AI가 만든 답변을 일회성 출력으로 끝내지 않고, 내 작업 흐름 안에 다시 넣을 수 있기 때문입니다.

### # 5) AI 분석은 대화가 아니라 재사용 가능한 결과여야 한다

AI를 쓰다 보면 좋은 답변을 얻을 때가 있습니다.

문제는 그 다음입니다.

그 답변을 어디에 둘 것인가.

채팅창에 그대로 두면 나중에 찾기 어렵습니다. 복사해서 문서에 붙이면 형식이 흐트러질 수 있습니다. 다시 정리하려면 또 다른 작업이 필요합니다.

Jisong Cloud 6.0에서 AI 분석은 이 문제를 해결하기 위한 기능입니다. 파일을 분석할 수 있고, 메모를 분석할 수 있고, 직접 입력한 텍스트도 분석할 수 있습니다.

분석 결과는 복사할 수 있습니다. Markdown으로 다운로드할 수 있습니다. PDF로 다운로드할 수도 있습니다. 필요하면 새 메모로 저장할 수 있습니다.

이 구조에서 AI는 단순한 대화 상대가 아닙니다. 작업 흐름 안에 들어온 분석 엔진에 가깝습니다.

AI가 답변을 만든다. 그 답변이 문서가 된다. 문서가 다시 메모가 된다. 메모가 다음 글이나 프로젝트의 재료가 된다.

이 흐름이 중요합니다.

AI를 잘 쓰는 것만큼이나, AI가 만든 결과를 어떻게 저장하고 다시 쓸 것인지가 중요하다고 생각합니다. 좋은 답변을 한 번 얻는 것보다, 그 답변이 내 작업 환경 안으로 들어오는 것이 더 중요할 때가 많습니다.

### # 6) 도구모음은 사소하지만 자주 쓰인다

Jisong Cloud에는 도구모음도 있습니다. 텍스트 클리너. MD to PDF. 글자 수 카운터. 정산 계산기. 저장소 상태. 접속 기록 관리.

이런 기능들은 하나하나만 보면 작습니다. 하지만 실제 작업에서는 이런 작은 기능이 자주 필요합니다.

글을 쓰다 보면 Markdown을 PDF로 뽑아야 합니다. 복사해 온 텍스트는 줄바꿈이 이상해서 정리해야 합니다. 브런치 글이나 자기소개서를 쓸 때는 글자 수를 확인해야 합니다. 모임이나 프로젝트 비용을 나눌 때는 정산이 필요합니다. 서버를 운영한다면 저장소 상태와 접속 기록도 확인해야 합니다.

이런 일들은 별도의 전문 앱을 켤 만큼 거창하지 않습니다. 하지만 매번 검색해서 다른 사이트에 들어가기는 귀찮습니다.

무엇보다 작업 흐름이 끊깁니다.

그래서 도구모음은 부가 기능처럼 보이지만, 실제 사용감에서는 꽤 중요한 역할을 합니다. 개인 작업대에는 커다란 장비만 필요한 것이 아닙니다. 가위, 포스트잇, 자, 클립 같은 작은 도구도 필요합니다.

디지털 작업대도 마찬가지입니다.

작은 도구들은 작업의 흐름을 끊기지 않게 만든다.

### # 7) 보안은 나중에 붙이는 기능이 아니다

개인용 클라우드를 만들 때 가장 조심해야 하는 부분은 보안입니다. 파일을 올리고, 메모를 쓰고, AI 분석까지 연결한다면 당연히 접근 제어가 필요합니다. 특히 개인 작업 환경은 “나만 쓰는 도구”라는 이유로 보안을 가볍게 보기 쉽습니다. 하지만 나만 쓰는 도구일수록 더 조심해야 합니다.

Jisong Cloud 6.0에서는 Cloudflare Access를 선택적으로 적용할 수 있고, 앱 내부에는 패스키와 계정 ID, 비밀번호 fallback 구조를 둡니다. 파일, 메모, AI 기능은 인증 경계 안에 두고, 접속 기록 조회나 삭제, 비밀번호 변경, 폴더 동기화, v6 parser 연동 같은 민감한 기능도 인증이 필요하도록 나눕니다. 물론 이것이 완벽한 보안을 의미하지는 않습니다.

보안은 “완성된 기능”이라기보다 계속 관리해야 하는 경계에 가깝습니다. 어디에서 접근을 막을 것인지. 어떤 API를 인증 뒤에 둘 것인지. 로그를 어떻게 남길 것인지. 사용량과 비용을 어떻게 제한할 것인지.

이런 것들이 모두 구조의 일부입니다. 클라우드 위에 개인 작업대를 올린다는 것은 편해지는 일이기도 하지만, 동시에 경계를 설계해야 하는 일이기도 합니다.

### # 8) 작업 흐름을 서버 위에 올린다는 것

Jisong Cloud 6.0을 만들면서 가장 분명해진 생각은 이것입니다. 개인 인프라는 저장 공간이 아니라 작업 방식의 외부화입니다.

내가 자주 하는 일을 관찰합니다. 어디에서 반복이 생기는지 봅니다. 어디에서 흐름이 끊기는지 봅니다. 그 끊기는 지점을 API, 저장소, UI, AI 분석, 문서 변환으로 연결합니다.

이렇게 하면 내 작업 방식이 눈에 보이기 시작합니다. 파일을 어디에 두는지. 메모가 언제 필요한지. AI 분석 결과가 어디에서 사라지는지. 어떤 도구를 반복해서 찾는지. 어떤 과정이 귀찮아서 미뤄지는지.

직접 만들면 불편함이 보입니다. 불편함이 보이면 구조를 바꿀 수 있습니다. 구조를 바꾸면 작업 방식도 조금씩 달라집니다. 그래서 Jisong Cloud는 단순한 개발 결과물이 아닙니다.

제가 반복해서 하던 일을 외부 구조로 꺼내 놓은 것입니다. 머릿속에서만 하던 흐름을 브라우저, 서버, 저장소, API, AI 분석으로 옮긴 것입니다. 조금 거창하게 말하면, 개인 작업 방식의 외골격에 가깝습니다.

### # 9) 개인용 클라우드는 저장소가 아니라 작업대다

처음에는 클라우드를 공간으로 생각했습니다. 파일을 올려두는 곳. 내 컴퓨터 밖에 있는 저장 공간. 필요할 때 꺼내 쓰는 보관함. 하지만 지금은 조금 다르게 생각합니다.

나에게 클라우드는 공간이라기보다 흐름입니다. 파일을 올리고, 읽고, 분석하고, 정리하고, 메모하고, 다시 문서로 만들고, 다음 작업에 재사용하는 흐름. Jisong Cloud 6.0은 그 흐름을 제 방식대로 묶어본 시도입니다.

중요한 것은 기능이 많다는 것이 아닙니다. 제가 반복해서 하는 일이 하나의 작업대 위에 올라왔다는 것입니다. Jisong Cloud v2가 반복을 발견한 실험이었다면, Jisong Cloud v6는 그 반복을 실제로 굴러가는 개인 작업 환경으로 옮긴 버전입니다. 개인용 클라우드는 저장소가 아니라 작업대입니다. 그리고 좋은 작업대는 일을 대신해주지는 않지만, 내가 하려던 일을 덜 잃어버리게 해줍니
