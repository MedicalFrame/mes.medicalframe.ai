## 사고 보조 장치, Antigravity IDE

Antigravity IDE: 코딩 도구가 아니라 사고 보조 장치 예전에는 코딩을 한다는 것이 직접 코드를 쓰는 일에 가깝다고 생각했습니다. 파일을 열고, 함수를 만들고, 에러를 고치고, 필요한 기능을 하나씩 구현했습니다. 물론 지금도 코드는 사람이 이해해야 합니다. 하지만 AI 코딩 도구를 쓰기 시작하면서 코딩의 중심이 조금 바뀌었습니다.

이제 중요한 것은 코드를 한 줄씩 직접 쓰는 능력만이 아닙니다. 무엇을 만들 것인지 정하고, 그 일을 몇 단계로 나눌지 정하고, 어떤 파일을 건드릴지 정하고, 어떤 함수가 어떤 책임을 가질지 정하고, 어떤 기준으로 완료 여부를 확인할지 정하는 일이 더 중요해졌습니다. 즉, 코딩은 문법의 문제가 아니라 작업을 구조화하는 문제에 가까워졌습니다. Antigravity IDE를 쓰면서 이 생각이 더 분명해졌습니다.

저에게 Antigravity IDE는 단순한 코딩 도구가 아닙니다. 생각을 파일, 함수, 작업 단위, 검증 기준으로 나누게 만드는 사고 보조 장치에 가깝습니다. Antigravity IDE 작업 화면

형태: AI 기반 개발 환경

사용 방식: ChatGPT에서 사고 구체화 → Codex용 프롬프트 작성 → Codex가 todo.md와 implementation plan 생성 → Antigravity IDE에서 Codex/Gemini 역할 분담 → 결과 검토 및 반영

연결 프로젝트: Jisong Cloud, CleanEMR, DiaFrame, PharmaFrame, Obsidian 관련 도구, 개인 웹앱 전반

핵심 구조: 문제 정의 → 작업 분해 → 문서화 → agent 실행 → 코드 수정 → 테스트/검토 → lessons 반영

주의: AI가 코드를 대신 책임지는 것이 아니라, 사람이 작업 범위와 검증 기준을 정해야 한다.

### # 1) 코딩 전에 먼저 생각을 구체화한다

요즘 저는 바로 IDE를 열고 코드를 치지 않습니다. 먼저 ChatGPT와 이야기합니다. 무엇을 만들 것인지, 왜 필요한지, 현재 문제가 무엇인지, 기능을 어느 범위까지 넣을지, 어떤 파일과 구조가 필요할지 먼저 말로 풀어봅니다. 이 과정은 단순한 잡담이 아닙니다.

생각을 구체화하는 단계입니다.

처음 머릿속에 있는 아이디어는 대개 흐립니다.

“이 기능 있으면 좋겠다.”

“여기 버그가 있는 것 같다.”

“이 프로젝트 구조를 좀 정리해야겠다.” 이 정도로는 AI agent에게 일을 시키기 어렵습니다. 사람에게도 애매한 말은 AI에게 더 위험합니다. 그래서 먼저 생각을 문장으로 바꿉니다.

무엇이 문제인지. 어떤 결과를 원하는지. 어디까지 바꾸면 되는지. 무엇은 건드리면 안 되는지. 완료 기준은 무엇인지.

이 단계에서 ChatGPT는 코딩 도구라기보다 생각을 정리하는 파트너에 가깝습니다. 생각이 어느 정도 정리되면, 그다음에는 Codex에게 줄 프롬프트로 바꿉니다.

### # 2) 좋은 프롬프트는 명령문이 아니라 작업 명세서다

AI에게 “이거 고쳐줘”라고 하면 정말 고쳐주기는 합니다.

문제는 어디까지 고칠지 모른다는 것입니다.

애매하게 시키면 AI는 지멋대로 판단합니다. 좋게 말하면 자율적이고, 나쁘게 말하면 망나니처럼 움직일 수 있습니다. 그래서 Codex에게 줄 프롬프트는 단순한 명령문이 아니라 작업 명세서에 가까워야 합니다.

현재 상황. 목표. 수정 범위. 건드리면 안 되는 파일. 예상 산출물. 테스트 방법. 완료 기준. 작업을 몇 턴으로 나눌지.

이런 것들이 들어가야 합니다.

명령을 잘해야 합니다.

AI가 똑똑해질수록 오히려 이 점이 더 중요해집니다. 애매한 지시를 주면 AI는 그럴듯한 방향으로 넓게 움직입니다. 하지만 프로젝트에서는 그 넓은 움직임이 위험할 수 있습니다. 원래 의도와 다른 파일을 고치거나, 이미 잡아둔 구조를 무너뜨리거나, 임시방편 코드를 추가하거나, 보안과 인증 경계를 건드릴 수 있습니다. 그래서 저는 AI에게 일을 시킬 때 먼저 작업을 좁힙니다.

“무엇을 할 것인가”만큼이나 “무엇을 하지 않을 것인가”를 적습니다. 좋은 프롬프트는 agent가 실행할 수 있는 작업 명세서가 된다.

### # 3) Codex는 todo.md와 implementation plan을 만든다

프롬프트를 Codex에게 넘기면 바로 코드를 고치게만 하지 않습니다.

먼저 계획을 세우게 합니다.

몇 턴으로 나눌지. 어떤 파일을 볼지. 어떤 순서로 수정할지. 어떤 테스트를 돌릴지. 어떤 위험이 있는지.

이 과정에서 todo.md와 implementation plan이 생깁니다. 저에게 이 문서들은 매우 중요합니다. AI agent가 지금 무엇을 하려는지 사람이 볼 수 있는 형태로 남기기 때문입니다.

그냥 “알아서 고쳐줘”라고 하면 AI가 무엇을 기준으로 움직이는지 알기 어렵습니다. 하지만 todo.md가 있으면 작업이 체크리스트로 보입니다. implementation plan이 있으면 변경 방향과 순서가 보입니다.

이렇게 되면 AI가 코드만 쓰는 것이 아니라 작업의 구조를 먼저 드러내게 됩니다. 그리고 이 구조를 사람이 검토할 수 있습니다. 여기서 방향이 틀렸다면 아직 코드를 많이 건드리기 전에 수정할 수 있습니다. AI 코딩에서 중요한 것은 AI가 얼마나 빨리 코드를 쓰는지가 아닙니다.

AI가 어떤 계획으로 움직이는지 사람이 볼 수 있어야 한다는 점입니다.

### # 4) Antigravity IDE에서는 agent를 역할별로 쓴다

Antigravity IDE를 쓰면서 편했던 점은 여러 AI 도구를 한 작업 환경 안에서 같이 쓸 수 있다는 점이었습니다. 저는 중요한 작업은 Codex에게 시키고, 상대적으로 간단한 작업은 Gemini에게 맡기는 식으로 씁니다. 예를 들어 구조를 크게 바꾸거나, 여러 파일을 건드리거나, 기존 설계와 충돌 가능성이 있는 작업은 Codex에게 맡깁니다. 반대로 작은 수정, 간단한 설명, 짧은 코드 조각, 문서 초안, 명확한 단일 작업은 Gemini에게 맡길 수 있습니다.

중요한 것은 AI를 하나의 거대한 지능으로 보지 않는 것입니다.

작업마다 적합한 역할이 다릅니다.

어떤 작업은 깊게 생각해야 합니다. 어떤 작업은 빠르게 처리하면 됩니다. 어떤 작업은 계획부터 세워야 하고, 어떤 작업은 이미 정해진 구조 안에서 작은 수정만 하면 됩니다. Antigravity IDE는 이런 역할 분담을 실제 개발 환경 안에서 하게 해줍니다. 그래서 IDE가 단순히 코드를 쓰는 창이 아니라 작업 지휘소처럼 느껴질 때가 있습니다.

코드를 직접 쓰는 곳이면서, agent에게 일을 나눠주는 곳이고, 산출물을 확인하는 곳이며, 프로젝트 구조를 다시 잡는 곳입니다.

### # 5) AGENTS.md, lessons.md, todo.md는 안전장치다

AI agent를 잘 쓰려면 말을 잘하는 것만으로는 부족합니다. 프로젝트 안에 기준 문서가 있어야 합니다.

저는 AGENTS.md, lessons.md, todo.md 같은 아티팩트를 중요하게 봅니다.

AGENTS.md는 agent가 프로젝트에서 지켜야 할 규칙을 정리하는 문서입니다. 어떤 구조를 따를지, 어떤 명령을 먼저 확인할지, 어떤 파일을 조심해야 할지, 어떤 방식으로 테스트해야 할지 기본 규칙을 알려줍니다.

lessons.md는 이전 작업에서 배운 점을 남기는 문서입니다. 어떤 실수를 했는지, 어떤 방식이 좋았는지, 다음에는 무엇을 피해야 하는지 프로젝트의 경험치를 쌓아둡니다.

todo.md는 지금 진행 중인 작업을 작은 단위로 쪼개는 문서입니다. 이 세 가지가 있으면 AI가 훨씬 덜 망나니처럼 움직입니다.

완전히 통제된다는 뜻은 아닙니다.

하지만 최소한 프로젝트 안에서 어떤 방향으로 움직여야 하는지 기준이 생깁니다. AI agent에게는 맥락이 필요합니다. 그리고 그 맥락은 채팅창 안에만 있으면 약합니다. 프로젝트 폴더 안에 문서로 남아 있어야 다음 작업에서도 다시 참조할 수 있습니다.

AI를 통제하는 가장 현실적인 방법은 매번 말로 붙잡는 것이 아니라 프로젝트 안에 규칙과 기록을 남겨두는 것입니다.

AGENTS.md, lessons.md, todo.md는 AI agent가 프로젝트 안에서 길을 잃지 않게 하는 안전장치다.

### # 6) 예전에는 직접 짰고, 그다음에는 복붙했다

AI IDE를 쓰기 전에는 대부분 직접 코드를 짰습니다. 필요한 기능이 있으면 직접 구현하고, 에러가 나면 직접 로그를 보고, 문서를 찾아보고, 하나씩 고쳤습니다. 그 다음 단계에서는 ChatGPT를 많이 썼습니다. 코드를 복사해서 붙여넣고, 에러 메시지를 넣고, 수정안을 받아서 다시 IDE에 붙였습니다.

이 방식도 꽤 유용했습니다.

하지만 한계가 있었습니다.

맥락이 자주 끊겼습니다.

ChatGPT는 전체 프로젝트 파일 구조를 계속 보고 있는 것이 아니었고, 제가 필요한 파일을 골라서 붙여넣어야 했습니다. 수정안이 나오면 다시 IDE로 옮겨야 했습니다. 여러 파일을 건드리는 작업에서는 어떤 파일에 어떤 변경을 해야 하는지 사람이 계속 이어붙여야 했습니다. 이 방식은 AI를 조언자로 쓰는 방식에 가까웠습니다.

지금은 조금 다릅니다.

ChatGPT에서 사고를 구체화하고, Codex용 프롬프트를 만들고, Codex가 계획을 세우고, Antigravity IDE 안에서 agent가 실제 파일을 보고 수정합니다. 이제 AI는 채팅창 밖으로 나와서 프로젝트 구조 안에서 움직입니다.

그만큼 편해졌습니다.

하지만 그만큼 더 조심해야 합니다.

### # 7) AI가 파일을 만질 수 있다는 것은 위험한 장점이다

AI IDE의 가장 큰 장점은 AI가 프로젝트 파일을 직접 보고 수정할 수 있다는 점입니다. 동시에 이것이 가장 큰 위험입니다. 채팅창에서 답변만 받을 때는 틀린 답을 받아도 바로 프로젝트가 망가지지는 않습니다. 하지만 IDE 안에서 agent가 파일을 직접 수정하면 잘못된 판단이 바로 코드 변경으로 이어질 수 있습니다. 그래서 AI IDE에서는 검토 구조가 더 중요합니다.

무엇을 바꿨는지 확인해야 합니다. 왜 바꿨는지 설명을 봐야 합니다. 테스트를 돌려야 합니다. diff를 봐야 합니다. 예상하지 않은 파일이 바뀌지 않았는지 확인해야 합니다.

AI가 코드를 생성했다고 해서 그 코드가 곧바로 내 코드가 되는 것은 아닙니다.

AI가 만든 것은 제안입니다.

사람이 받아들이고 검토한 뒤에야 프로젝트의 일부가 됩니다. 이 기준을 놓치면 AI IDE는 좋은 도구가 아니라 프로젝트를 엉망으로 만드는 빠른 손이 될 수 있습니다.

### # 8) AI IDE를 잘 쓰려면 사람이 더 구조적이어야 한다

AI IDE를 쓰면 코딩이 쉬워집니다. 하지만 역설적으로 사람은 더 구조적이어야 합니다. 예전에는 내가 직접 코드를 짰기 때문에 내 머릿속에 어느 정도 구조가 있었습니다. 이제는 AI가 많은 부분을 대신 움직일 수 있습니다.

그러면 사람이 구조를 놓치기 쉽습니다. 어떤 파일이 어떤 책임을 갖는지. 이 함수는 왜 필요한지. 이 변경은 기존 설계와 맞는지. 테스트는 충분한지. 보안 경계는 유지되는지. 지금 수정이 임시방편인지, 장기 구조에 맞는지.

이 질문을 계속 사람이 잡아야 합니다. AI에게 맡길수록 사람은 더 큰 구조를 봐야 합니다. 명령을 잘해야 하고, 작업을 나눠야 하고, 산출물을 검토해야 하고, 이전 실수를 lessons로 남겨야 합니다. 잘 쓰면 AI IDE는 굉장히 좋은 도구입니다. 하지만 잘 쓰는 방법을 알아야 합니다.

### # 9) IDE는 코딩 도구가 아니라 사고 보조 장치다

Antigravity IDE를 쓰면서 IDE에 대한 생각이 조금 바뀌었습니다. 예전의 IDE는 코드를 쓰는 곳이었습니다. 하지만 지금의 AI IDE는 코드를 쓰는 곳이면서 동시에 작업을 나누고, agent를 배치하고, 계획을 확인하고, 변경을 검토하고, 프로젝트의 구조를 관리하는 곳입니다. 코딩을 잘한다는 것은 문법을 많이 아는 것만은 아닙니다.

문제를 파일로 나누고, 함수로 나누고, 책임으로 나누고, 테스트 가능한 단위로 나누는 능력입니다. AI IDE는 이 과정을 빠르게 도와줍니다.

하지만 대신해주지는 않습니다.

결국 중요한 것은 어떤 질문을 코드의 형태로 바꿀 것인가입니다.

질문이 흐리면 코드도 흐려집니다.

구조가 없으면 AI도 구조 없이 움직입니다. 반대로 사람이 문제를 잘 나누고, 규칙을 문서로 남기고, 작업 범위를 좁히고, 검증 기준을 명확히 하면 AI IDE는 강력한 사고 보조 장치가 됩니다. 저에게 Antigravity IDE는 코딩을 대신해주는 도구라기보다 생각을 코드 구조로 바꾸는 작업 환경입니다. 코딩 도구가 아니라, 사고 보조 장치에 가깝습니다.
