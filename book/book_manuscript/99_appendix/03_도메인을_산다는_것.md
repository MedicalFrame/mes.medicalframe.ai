## 도메인을 산다는 것

처음에는 앱을 하나씩 만들었습니다. 필요한 기능이 있으면 만들고, 작은 실험이 떠오르면 레포를 만들고, 어떤 프로젝트는 Cloud Run에 올리고, 어떤 프로젝트는 GitHub Pages에 올렸습니다.

처음에는 그것으로 충분했습니다.

작동하면 됐고, 내가 쓸 수 있으면 됐고, 주소가 조금 길어도 크게 상관없었습니다. 하지만 프로젝트가 하나씩 늘어나면서 문제가 생겼습니다.

앱은 있는데 어디에 있는지 헷갈렸습니다. 레포는 있는데 무엇을 위한 것인지 다시 설명해야 했습니다. 포트폴리오는 따로 있고, 사진은 또 다른 곳에 있고, 개인 클라우드는 별도의 주소에 있었습니다. 작업은 분명히 쌓이고 있었는데, 그 작업들이 하나의 공간 안에 들어와 있지는 않았습니다.

그때 도메인을 샀습니다.

jisong.dev

처음에는 그냥 주소라고 생각했습니다. 하지만 막상 도메인을 사고 나니, 이것이 단순한 주소 이상의 의미를 가진다는 것을 알게 되었습니다. 도메인을 산다는 것은 인터넷 위에 내 작업의 좌표계를 하나 만드는 일이었습니다.

주의: 도메인은 자기소개용 장식이 아니라, 작업과 정체성을 정렬하는 개인 인프라의 일부

### # 1) 주소가 생기면 경계가 생긴다

도메인을 사기 전에도 프로젝트는 있었습니다. GitHub 레포가 있었고, 배포된 앱이 있었고, 개인적으로 쓰는 도구도 있었습니다. 하지만 그때의 프로젝트들은 조금 흩어져 있었습니다. 각자 다른 주소를 가지고 있었고, 어떤 것은 임시 배포 링크에 가까웠고, 어떤 것은 로컬에서만 돌아갔고, 어떤 것은 이름은 있지만 외부에서 접근하기 어려웠습니다.

주소가 없다고 해서 프로젝트가 없는 것은 아닙니다. 하지만 주소가 없으면 경계가 흐립니다. 이것이 어디에 속한 작업인지, 어디까지가 하나의 프로젝트인지, 어떤 이름으로 기억해야 하는지 불분명해집니다. 도메인은 그 흐릿한 경계에 선을 긋습니다.

jisong.dev 아래에 놓인다는 것은 이 작업이 내 개인 개발 공간의 일부라는 뜻이 됩니다.

photo.jisong.dev라는 이름이 붙는 순간, 사진은 그냥 이미지 파일 묶음이 아니라 하나의 아카이브가 됩니다. Jisong Cloud에 주소가 붙으면, 그것은 더 이상 로컬에서만 쓰는 실험이 아니라 인터넷 위에서 접근 가능한 개인 작업대가 됩니다.

주소는 단순한 링크가 아닙니다.

작업의 경계입니다.

### # 2) 이름이 생기면 실험은 공간이 된다

처음 만드는 프로젝트는 대개 임시적입니다. 일단 만들어봅니다. 작동하는지 확인합니다. 필요하면 고칩니다. 별로면 버립니다.

이 단계에서는 이름이 그렇게 중요하지 않습니다. 하지만 어느 순간부터는 이름이 필요해집니다.

반복해서 쓰기 시작할 때. 다른 프로젝트와 연결되기 시작할 때. 다시 돌아와서 고치기 시작할 때. 누군가에게 설명해야 할 때.

이름이 생기면 프로젝트의 상태가 달라집니다. 그전까지는 “예전에 만든 그거”였던 것이 이제는 하나의 공간이 됩니다. jisong.dev는 그런 의미에서 제 작업의 루트 디렉토리 같은 이름입니다.

그 아래에 포트폴리오가 있고, 사진이 있고, 개인용 클라우드가 있고, 앞으로 생길 다른 프로젝트가 들어올 수 있습니다. 이름은 단순히 보기 좋게 붙이는 라벨이 아닙니다. 이름은 작업을 다시 찾을 수 있게 하고, 다른 작업과 연결할 수 있게 하고, 외부에 설명할 수 있게 만듭니다. 프로젝트가 이름을 얻는 순간, 그것은 임시 실험에서 하나의 장소가 됩니다.

### # 3) 도메인은 작업의 좌표계다

좌표계가 없으면 위치를 설명하기 어렵습니다. 어떤 점이 어디에 있는지 말하려면 기준이 필요합니다.

도메인도 비슷합니다.

인터넷 위에는 수많은 링크가 있습니다. GitHub 주소도 있고, Cloud Run 주소도 있고, 임시 배포 주소도 있고, 브런치 주소도 있고, 사진 공유 링크도 있습니다. 각 링크는 존재하지만, 그 자체로는 하나의 세계를 만들지 않습니다. 도메인은 이 흩어진 링크들에 기준점을 줍니다.

jisong.dev라는 기준점이 생기면, 각 프로젝트는 그 기준점에 대해 위치를 가질 수 있습니다.

메인 포트폴리오는 루트에 둡니다. 사진은 photo.jisong.dev에 둡니다. 개인 클라우드는 별도의 하위 도메인이나 연결 주소에 둡니다. 다른 프로젝트들도 필요하면 각각의 이름을 가질 수 있습니다.

이렇게 되면 작업들이 단순히 나열되지 않습니다.

정렬됩니다.

도메인을 산다는 것은 웹 주소 하나를 소유하는 일이기도 하지만, 동시에 내 작업들의 좌표계를 만드는 일입니다. 어디에 무엇을 둘 것인가. 무엇을 공개하고, 무엇을 닫아둘 것인가. 어떤 프로젝트는 포트폴리오로 보이고, 어떤 프로젝트는 개인 도구로 남길 것인가.

이 질문들이 도메인 위에서 정리됩니다.

### # 4) photo.jisong.dev는 사진 저장소가 아니다

사진도 비슷했습니다.

처음에는 그냥 찍었습니다.

좋아 보이는 장면을 찍고, 빛이 예쁘면 찍고, 선과 구조가 마음에 들면 찍었습니다.

사진 파일은 계속 쌓였습니다.

하지만 파일로만 남아 있으면 사진은 금방 흩어집니다. 휴대폰 앨범 안에 있고, 외장하드 어딘가에 있고, 가끔 SNS에 올리고, 다시 찾으려면 시간이 걸립니다.

photo.jisong.dev라는 이름을 붙이면 상황이 조금 달라집니다. 사진은 단순한 파일 묶음이 아니라 하나의 아카이브가 됩니다.

무엇을 찍는 사람인지, 어떤 분위기를 좋아하는지, 어떤 구조를 보는지 조금 더 분명해집니다. 저에게 사진은 감성의 기록이기도 하지만, 동시에 구조를 보는 일이기도 합니다.

빛. 선. 공간. 여백. 반복. 균형. 이런 것들을 모아두는 공간으로서 photo.jisong.dev는 단순한 사진 저장소가 아니라 시각적 사고의 한 영역이 됩니다. 도메인은 여기서도 같은 역할을 합니다.

흩어진 파일에 이름을 주고, 그 이름 아래에 경계를 만들고, 경계 안에서 다시 정렬하게 합니다.

photo.jisong.dev는 사진 파일 묶음이 아니라, 시각적 사고를 정렬하는 공간이다.

### # 5) 개인 클라우드는 작업대가 되고, 도메인은 문패가 된다

Jisong Cloud를 만들면서도 비슷한 생각을 했습니다. 처음에는 파일 업로드와 메모장, 작은 도구를 모은 개인용 앱이었습니다. 하지만 시간이 지나면서 Jisong Cloud는 단순 저장소에서 작업대로 바뀌었습니다. 파일을 올리고, 메모를 쓰고, AI로 분석하고, Markdown을 PDF로 바꾸고, 결과를 다시 저장하는 흐름이 생겼습니다.

이런 작업대에도 문패가 필요합니다. 내가 어디로 들어가야 하는지, 이 작업대가 무엇을 위한 공간인지, 어디까지가 개인 도구이고 어디부터가 공개 프로젝트인지 정리해주는 이름이 필요합니다.

도메인은 그 문패 역할을 합니다.

주소가 생기면 접속이 쉬워집니다.

하지만 그것보다 더 중요한 것은 작업을 인식하는 방식이 달라진다는 점입니다. 이제 이것은 “어딘가에 배포된 앱”이 아니라 내 작업 환경의 한 부분이 됩니다. 도메인은 프로젝트를 더 그럴듯하게 포장하는 장식이 아닙니다. 프로젝트가 어디에 속하는지를 알려주는 경계입니다.

### # 6) 공개할 것과 닫아둘 것을 나누게 된다

도메인을 갖는다는 것은 무엇을 보여줄지 정하는 일이기도 합니다. 모든 프로젝트를 공개할 필요는 없습니다.

어떤 것은 포트폴리오로 보여줄 수 있습니다. 어떤 것은 설명 글로만 남겨도 됩니다. 어떤 것은 로그인 뒤에 숨겨야 합니다. 어떤 것은 아예 로컬에만 두는 편이 낫습니다.

개인 인프라에서는 이 구분이 중요합니다. 특히 파일, 메모, AI 분석, 개인 기록이 들어가는 도구라면 주소가 생겼다고 해서 모두에게 열어두면 안 됩니다. 도메인은 공개의 수단이지만, 동시에 경계를 설계하게 만드는 장치이기도 합니다.

어떤 하위 도메인은 공개 페이지로 둡니다. 어떤 하위 도메인은 인증 뒤에 둡니다. 어떤 프로젝트는 GitHub에 공개하고, 어떤 프로젝트는 private repository로 남깁니다. 이런 구분을 하다 보면 내 작업의 성격도 더 분명해집니다.

보여주기 위한 작업. 실제로 쓰기 위한 작업. 실험 중인 작업. 아직 이름 붙이기 전의 작업.

도메인은 이 모든 것을 한 번에 공개하라는 뜻이 아닙니다. 오히려 무엇을 공개하고, 무엇을 닫아둘지 정리하게 만듭니다.

### # 7) 도메인은 정체성을 과시하는 도구가 아니다

개인 도메인을 산다고 하면 조금 자기 브랜딩처럼 보일 수 있습니다.

틀린 말은 아닙니다.

도메인은 분명히 외부에 보이는 이름입니다. 누군가에게 링크를 줄 수 있고, 포트폴리오에 적을 수 있고, 프로젝트를 소개할 때 사용할 수 있습니다. 하지만 제가 도메인을 산 이유는 정체성을 크게 보여주기 위해서만은 아니었습니다. 오히려 정체성을 정리하기 위해서였습니다.

의대생. 엔지니어. 글 쓰는 사람. 사진 찍는 사람. 개인 도구를 만드는 사람. 의료 데이터를 구조화하고 싶은 사람.

이런 것들은 따로 보면 조금 흩어져 보입니다. 하지만 제 안에서는 완전히 따로 떨어져 있지 않습니다.

공통점은 구조를 본다는 것입니다.

의학에서도 구조를 보고, 코드에서도 구조를 보고, 사진에서도 구조를 보고, 글에서도 구조를 봅니다. 도메인은 이 흩어진 작업들을 하나의 이름 아래에 느슨하게 묶어줍니다. 나를 설명하기 위한 거창한 선언문이 아니라, 내 작업들이 모이는 좌표계로서 작동합니다.

### # 8) 이름을 붙인다는 것은 책임을 지는 일이다

작업에 이름을 붙이면 조금 부담이 생깁니다.

그전까지는 그냥 실험이었습니다.

잘 안 되면 지워도 되고, 주소가 바뀌어도 상관없고, 설명이 부족해도 크게 문제 되지 않았습니다. 하지만 도메인 아래에 올리면 조금 더 신경 쓰게 됩니다.

첫 화면은 괜찮은지. 링크는 깨지지 않는지. 무엇을 위한 공간인지 설명되는지. 보안 경계는 적절한지. 너무 오래된 프로젝트를 그대로 두고 있지는 않은지.

이 부담이 나쁘지만은 않습니다.

오히려 작업을 관리하게 만듭니다.

도메인은 프로젝트를 완성시켜주지는 않습니다. 하지만 프로젝트를 방치하기 어렵게 만듭니다. 이름이 생긴 작업은 다시 돌아가서 고칠 수 있는 대상이 됩니다. 그래서 도메인을 산다는 것은 단순히 주소를 결제하는 일이 아닙니다.

내 작업에 이름을 붙이고, 그 이름 아래에 어느 정도 책임을 지겠다는 뜻이기도 합니다.

### # 9) 도메인을 산다는 것

도메인을 산다는 것은 인터넷 위에 내 이름을 하나 더 올리는 일이 아닙니다. 그보다는 흩어진 작업을 하나의 기준점 아래에 정렬하는 일에 가깝습니다.

파일은 저장소에 있을 수 있습니다. 코드는 GitHub에 있을 수 있습니다. 앱은 Cloud Run에 있을 수 있습니다. 사진은 별도의 페이지에 있을 수 있습니다. 글은 브런치에 있을 수 있습니다. 하지만 그 모든 것이 완전히 흩어진 상태로만 남아 있으면 작업은 쌓이지만 방향은 흐려집니다.

도메인은 그 작업들 사이에 좌표계를 만듭니다. 무엇을 어디에 둘 것인지. 어떤 이름으로 부를 것인지. 어디까지 공개할 것인지. 무엇을 내 작업의 일부로 받아들일 것인지.

이런 질문에 답하게 만듭니다.

저에게 jisong.dev는 단순한 주소가 아닙니다.

작업의 기준점입니다.

photo.jisong.dev는 단순한 사진 저장소가 아닙니다. 시각적 사고를 모아두는 공간입니다.

Jisong Cloud는 단순한 웹앱이 아닙니다. 반복되는 작업을 올려둔 개인 작업대입니다. 도메인을 산다는 것은 이 모든 것을 하나의 이름 아래에 배치하는 일입니다. 기술적 소유물이 아니라, 작업의 좌표계를 만드는 일. 그리고 좌표계가 생기면 내가 어디에 서 있는지도 조금 더 잘 보입니다.
